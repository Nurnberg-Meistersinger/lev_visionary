# The Visionary

The Visionary — AI-платформа для автоматической агрегации и анализа контента: собирает информацию из RSS-источников и X/Twitter, обрабатывает через Claude API и отправляет дайджесты в Telegram.

---

## Структура проектов

| Проект | Источник | Что делает |
| --- | --- | --- |
| `project_a16z` | a16z Daily Newsletter | Скачивает свежие статьи, генерирует TL;DR + summary |
| `project_techcrunch_startup` | TechCrunch Startups | TL;DR + summary для публикаций о стартапах |
| `project_techcrunch_venture` | TechCrunch Venture | TL;DR + summary для секции Venture |
| `project_twitter` | X/Twitter | Подбирает важные твиты по группам аккаунтов, анализирует через LLM |

---

## Быстрый старт

1. Python 3.12+ и виртуальное окружение:
   ```bash
   python -m venv .the_vis && source ./.the_vis/bin/activate
   ```

2. Установка зависимостей:
   ```bash
   pip install -r requirements.txt
   ```

3. Заполните `.env` (см. раздел [Конфигурация](#конфигурация-env))

4. Запуск:
   ```bash
   python visionary.py run --all
   ```

## Конфигурация `.env`

```env
# Claude API (Anthropic)
LLM_MODEL=claude-haiku-4-5-20251001
LLM_API_KEY=sk-ant-api03-...

# Telegram
TELEGRAM_TOKEN=your_bot_token
TELEGRAM_CHAT_ID=-100xxxxxxxxxx

# Twitter (для project_twitter)
TWITTER_USERNAME=your_twitter_handle
TWITTER_EMAIL=your_email@example.com
TWITTER_PASSWORD=your_password

# Bot admin (для управления через Telegram)
ADMIN_USER_ID=your_telegram_user_id

# Расписание автоматических дайджестов (опционально)
SCHEDULE_ENABLED=true
SCHEDULE_TIME=09:00
SCHEDULE_TIMEZONE=Europe/Moscow
```

Часовые пояса: `Europe/Moscow`, `UTC`, `Europe/Berlin`, `Asia/Tbilisi` и т.д. (стандарт IANA).

### Получение ключей

- **Claude API**: регистрация на [console.anthropic.com](https://console.anthropic.com), пополнение баланса от $5
- **Telegram**:
  - Создать бота через [@BotFather](https://t.me/BotFather), получить `TELEGRAM_TOKEN`
  - Создать канал, добавить бота как администратора, получить `TELEGRAM_CHAT_ID` (например, `-100xxxxxxxxxx`)
  - Узнать свой `ADMIN_USER_ID` через [@userinfobot](https://t.me/userinfobot)
- **Twitter**: credentials от аккаунта (рекомендуется отдельный аккаунт)

## LLM и промпты

Все проекты используют Claude API (`anthropic`). Промпты:

- `project_*/summarizer.py` — TL;DR и summary для статей
- `project_twitter/llm_filter.py` — анализ и отбор важных твитов

### Критерии отбора твитов

LLM оценивает каждый твит по шкале 0–10:
- Конкретные новости: анонсы, сделки, партнёрства, релизы
- Инсайдерская информация от руководителей компаний
- Аналитика с цифрами и метриками
- Стратегические прогнозы экспертов

Не отбираются: ретвиты без комментариев, личные мнения без фактов, шутки, мемы.

# Аналитический паспорт решения

> Раздел для аналитиков и авторов документации. Содержит всю информацию, необходимую для подготовки проектного отчёта по шаблону.

---

## 1. Контекст задачи

**Для кого:** индивидуальный пользователь или небольшая команда, которые следят за новостями в сфере технологий, венчурного рынка, стартапов и крипто/блокчейна.

**Бизнес-контекст:** ежедневный мониторинг 10+ источников занимает 1–2 часа: нужно открыть несколько сайтов, пролистать ленту Twitter, выбрать релевантное и прочитать. Значительная часть контента нерелевантна или низкокачественна.

**Решаемая задача:** автоматически агрегировать, фильтровать и резюмировать контент из целевых источников, доставляя готовый дайджест в Telegram.

---

## 2. Проблема и цель

**Боль / проблема:**
- Ручной мониторинг новостей занимает 60–120 минут/день
- Высокий «шум»: большинство публикаций не несут значимой информации
- Источники разрознены: RSS-фиды, Twitter, рассылки — у каждого свой интерфейс
- Нет единого места для потребления контента

**Цель:** сократить время на мониторинг до 10–15 минут/день за счёт автоматической агрегации и LLM-суммаризации.

**KPI:**
- Среднее время чтения дайджеста: ≤ 15 мин/день
- Охват ключевых источников: 4+ источника ежедневно
- Отказоустойчивость: ≥ 95% успешных ежедневных запусков
- Стоимость LLM: ≤ $10/месяц

---

## 3. Ограничения

| Категория | Описание |
|---|---|
| **Безопасность** | Credentials (Twitter, Telegram, Claude API) хранятся локально в `.env`, не коммитятся в репозиторий |
| **Twitter ToS** | Автоматический сбор твитов через `twikit` может нарушать правила X/Twitter при высоких частотах запросов |
| **Rate limits** | Twitter ограничивает частоту запросов — задержки 90 сек между наборами обязательны |
| **Бюджет** | Claude Haiku — минимальная стоимость среди Claude-моделей; при текущем объёме ~$3–6/мес |
| **Инфраструктура** | Требуется VPS для 24/7 работы бота; достаточно минимального сервера (1 CPU, 512 MB RAM) |
| **Зависимость от внешних API** | Сбои Anthropic API или Telegram API прерывают дайджест |

## 4. Архитектура

### 4.1 Схема потока данных

```
Источники данных
     │
     ├── RSS: a16z Newsletter ──────────────────────┐
     ├── RSS: TechCrunch Startups ──────────────────┤
     ├── RSS: TechCrunch Venture ───────────────────┤
     └── X/Twitter (twikit scraping) ───────────────┤
                                                     │
                                            Слой сбора данных
                                          (rss_reader / tc_reader /
                                            fetcher.py + twikit)
                                                     │
                                            Дедупликация
                                          (processed_posts.json /
                                           processed_news.json)
                                                     │
                                            AI-слой (Claude API)
                                          ┌──────────┴──────────┐
                                    Суммаризация          LLM-фильтрация
                                   (summarizer.py)      (llm_filter.py)
                                    TL;DR + summary       топ-N твитов
                                                     │
                                            Доставка результата
                                          (telegram_sender.py)
                                                     │
                                         Telegram-канал / личка
```

### 4.2 Источники данных

| Источник | Метод | Формат | Модуль |
|---|---|---|---|
| a16z Newsletter | feedparser (RSS) | Atom/RSS XML | `project_a16z/rss_reader.py` |
| TechCrunch Startups | feedparser (RSS) | Atom/RSS XML | `project_techcrunch_startup/tc_reader.py` |
| TechCrunch Venture | feedparser (RSS) | Atom/RSS XML | `project_techcrunch_venture/tc_reader.py` |
| X/Twitter | twikit (web scraping) | JSON (внутренний API Twitter) | `project_twitter/fetcher.py` |

### 4.3 Слой автоматизации

- **Планировщик**: встроенный job_queue библиотеки `python-telegram-bot[job-queue]` — ежедневный запуск в заданное время (IANA timezone)
- **Триггеры**: расписание (`SCHEDULE_ENABLED`) + ручные команды через Telegram-бота (`/digest`)
- **Оркестратор**: `visionary.py run --all` для запуска из CLI
- **Задержки**: 60 сек между RSS-проектами, 90 сек между Twitter-наборами (защита от rate limit)

### 4.4 AI-слой

| Задача | Модуль | Описание |
|---|---|---|
| Суммаризация статей | `project_*/summarizer.py` | TL;DR (2 предложения), summary (5–7 предложений), 4–5 bullet points |
| Ранжирование статей | `project_*/article_ranker.py` | Оценка релевантности перед суммаризацией |
| Фильтрация твитов | `project_twitter/llm_filter.py` | Оценка важности 0–10, выбор топ-N, генерация summary + insights |

Модель: `claude-haiku-4-5-20251001` (настраивается через `LLM_MODEL` в `.env`).

### 4.5 Хранение данных

| Данные | Формат | Расположение | Назначение |
|---|---|---|---|
| Обработанные URL статей | JSON (список строк) | `project_*/processed_news.json` | Дедупликация — статья обрабатывается только раз |
| Обработанные ID твитов | JSON (список строк) | `project_twitter/processed_posts.json` | Дедупликация твитов |
| Кастомные наборы аккаунтов | JSON (объект) | `project_twitter/account_sets/*.json` | Описание набора: name, accounts, custom flag |
| Twitter-сессия | JSON (cookies) | `project_twitter/cookies.json` | Авторизация без повторного логина |
| Конфигурация | `.env` | корень проекта | Все secrets и настройки |

### 4.6 Каналы доставки

- **Telegram-канал**: основной канал доставки дайджестов (`TELEGRAM_CHAT_ID`)
- **Личка администратора**: отчёт о плановом запуске (`ADMIN_USER_ID`)
- **Telegram-бот**: интерактивное управление через команды

### 4.7 Роли и доступ

| Роль | Доступ | Инструмент |
|---|---|---|
| Администратор | Все команды бота, получает отчёты | Telegram (`ADMIN_USER_ID`) |
| Читатель канала | Только чтение дайджестов | Telegram-канал |
| Система | Автоматические запуски по расписанию | job_queue на VPS |

## 5. Описание сценариев

### Сценарий 1: Ежедневный автодайджест по расписанию

**Описание:** каждый день в заданное время бот запускает все проекты последовательно и отправляет дайджесты в канал + отчёт администратору.

**Входные данные:** нет (триггер — время по IANA timezone)

**Логика:**
1. Job scheduler срабатывает в `SCHEDULE_TIME`
2. Запускаются RSS-проекты: a16z → задержка 60 сек → TechCrunch Startup → задержка 60 сек → TechCrunch Venture
3. Запускаются Twitter-наборы: каждый с задержкой 90 сек
4. Ошибки логируются, не прерывают остальные проекты
5. Итоговый отчёт отправляется администратору

**Выход:** серия Telegram-сообщений в канале + итоговый отчёт в личку администратора

---

### Сценарий 2: Дайджест по команде из Telegram

**Описание:** администратор вручную запускает нужный дайджест через `/digest <project>`.

**Входные данные:** Telegram-команда `/digest palantir` (или другое имя проекта)

**Логика:**
1. Бот проверяет `user_id == ADMIN_USER_ID` (иначе — `⛔ Доступ запрещён`)
2. Парсит аргумент: Twitter-набор или другой проект
3. Отправляет статус `⏳ Запускаю: palantir...`
4. Запускает `process_set_async(set_name, top_n=10)`
5. По завершении обновляет статус: `✅ Дайджест Palantir отправлен в канал!`

**Выход:** дайджест в канале + обновлённое статусное сообщение в чате

---

### Сценарий 3: RSS-суммаризация (a16z / TechCrunch)

**Описание:** агент читает RSS-ленту, фильтрует новые статьи, парсит HTML, суммаризирует через Claude.

**Входные данные:** RSS-фид URL (настроен в `config.py` каждого проекта)

**Логика:**
1. `rss_reader.py` / `tc_reader.py` — получить список статей из RSS
2. Проверить `processed_news.json` — пропустить уже обработанные
3. `article_parser.py` — извлечь текст из HTML страницы
4. `article_ranker.py` — оценить релевантность (опционально)
5. `summarizer.py` → Claude API — сгенерировать TL;DR + summary + bullet points
6. `telegram_sender.py` — отформатировать и отправить в канал
7. Сохранить URL в `processed_news.json`

**Выход:** Telegram-сообщение вида:
```
📰 Заголовок статьи

💡 TL;DR: два предложения о главном

📋 Summary: 5-7 предложений с ключевыми идеями

• Пункт 1
• Пункт 2
• ...

🔗 Читать полностью
```

---

### Сценарий 4: Twitter-дайджест с LLM-фильтрацией

**Описание:** собираем твиты из набора аккаунтов, отбираем важные через LLM, отправляем дайджест.

**Входные данные:** JSON-файл набора аккаунтов (`account_sets/palantir.json`)

**Логика:**
1. `fetcher.py` — авторизоваться через twikit (cookies), получить твиты каждого аккаунта за последние N часов
2. Исключить уже обработанные (post_tracker.py)
3. `llm_filter.py` → Claude API — оценить каждый твит (0–10), выбрать топ-N
4. `telegram_sender.py` — сформировать и отправить дайджест

**Выход:** Telegram-сообщение со списком отобранных твитов:
```
🐦 Palantir Digest

1. Контракт Palantir с NATO на $500M
   📊 Summary: ...
   • Факт 1
   • Факт 2
   🔗 twitter.com/...

2. ...
```

---

### Сценарий 5: Управление кастомными наборами аккаунтов

**Описание:** администратор создаёт, редактирует и удаляет кастомные наборы Twitter-аккаунтов через бота.

**Команды и логика:**

| Команда | Действие | Результат |
|---|---|---|
| `/newset crypto elonmusk naval` | Создаёт `account_sets/crypto.json` с `"custom": true` | Подтверждение + список аккаунтов |
| `/addto crypto pmarca` | Добавляет `pmarca` в существующий файл | Список добавленных + итого |
| `/delset crypto` | Удаляет файл (только кастомные) | Подтверждение удаления |
| `/digest crypto` | Запускает дайджест для нового набора | Дайджест в канале |

## 6. Промпт-инжиниринг

### 6.1 Промпт для суммаризации статей (`summarizer.py`)

**Тип:** zero-shot с жёстким форматом вывода (JSON)

```
Ты — профессиональный редактор технологических новостей. Проанализируй статью и верни ТОЛЬКО JSON:

{
  "tldr": "2 предложения на русском (макс 200 символов)",
  "summary": "5-7 предложений на русском с ключевыми идеями",
  "bullet_points": [
    "Пункт 1 (макс 100 символов)",
    ...
  ]
}

ТРЕБОВАНИЯ:
- TLDR: ровно 2 предложения, максимум 200 символов
- Summary: 5-7 предложений с ключевыми идеями
- Bullet points: строго 4-5 пунктов, каждый до 100 символов
- Весь текст ТОЛЬКО на русском языке
- БЕЗ markdown блоков
- ТОЛЬКО валидный JSON объект!

ARTICLE: {текст статьи}
```

**Параметры:** `max_tokens=2048`, `temperature` — по умолчанию модели

---

### 6.2 Промпт для LLM-фильтрации твитов (`llm_filter.py`)

**Тип:** few-shot с примерами хороших/плохих ответов, жёсткий JSON-вывод

```
Ты — аналитический ИИ, который изучает твиты в области технологий, стартапов и блокчейна.

1. Оценить важность каждого твита по шкале 0–10
   Критерии важности:
   - Конкретные новости: анонсы продуктов, сделки, партнёрства, релизы
   - Инсайдерская информация: данные от руководителей компаний
   - Аналитика с цифрами: статистика, метрики, финансовые показатели
   - Стратегические прогнозы от экспертов индустрии

   НЕ выбирай: ретвиты без комментариев, личные мнения без фактов, шутки, мемы.

2. Выбрать ровно {top_n} самых важных твитов (importance >= 6).

3. По каждому выбранному твиту выдать (ВСЁ НА РУССКОМ):
   • summary: 2–3 предложения (до 300 символов)
   • insights: 2–3 ключевых факта (каждый до 100 символов)
   • short_tldr: главная суть в одной фразе (до 100 символов)

     ХОРОШИЕ примеры short_tldr:
     - "Контракт Palantir с NATO на $500M"
     - "Запуск AIP для коммерческих клиентов в Европе"

4. Вернуть строго JSON: { "important": [ {...} ] }
```

**Параметры:** `max_tokens=4096`, `temperature=0` (детерминированный результат)

## 7. Данные и модель данных

### 7.1 Хранимые данные

**`processed_news.json`** (project_a16z, project_techcrunch_*)
```json
["https://a16z.com/article-1", "https://techcrunch.com/article-2"]
```
Список URL уже обработанных статей. Обновляется после каждой успешной обработки.

**`processed_posts.json`** (project_twitter)
```json
["1234567890", "9876543210"]
```
Список ID твитов. Предотвращает повторную обработку.

**`account_sets/*.json`** (project_twitter)
```json
{
  "name": "Palantir",
  "description": "Руководство Palantir и близкие сообщества",
  "custom": false,
  "accounts": [
    {"handle": "PalantirTech", "name": "Palantir Technologies"},
    {"handle": "ssankar",      "name": "Shankar Sankar, CTO"}
  ]
}
```
Флаг `"custom": true` — признак кастомного набора, созданного через бота.

**`cookies.json`** (project_twitter)
Twitter-сессия twikit. Используется для авторизации без повторного ввода credentials.

### 7.2 Политика хранения данных

- Все данные хранятся локально на сервере, в директории проекта
- `processed_*.json` — не очищаются автоматически; при необходимости сбросить — удалить файл вручную
- `cookies.json` — пересоздаётся автоматически при истечении сессии
- Credentials в `.env` никогда не коммитятся в git (`.gitignore`)
- Дайджесты не хранятся — отправляются в Telegram и более не сохраняются

## 8. Этика и безопасность

### 8.1 Чувствительные данные

| Данные | Где хранятся | Уровень чувствительности |
|---|---|---|
| `LLM_API_KEY` (Anthropic) | `.env` | Высокий — утечка = несанкционированные расходы |
| `TELEGRAM_TOKEN` | `.env` | Высокий — утечка = перехват управления ботом |
| `TWITTER_PASSWORD` | `.env` | Высокий — утечка = компрометация аккаунта |
| `ADMIN_USER_ID` | `.env` | Средний — определяет, кто управляет ботом |
| `cookies.json` | файл на сервере | Средний — активная сессия Twitter |

### 8.2 Меры защиты

- `.env` добавлен в `.gitignore` — не коммитится в репозиторий
- Доступ к командам бота — только для `ADMIN_USER_ID` (декоратор `@admin_only`)
- Удаление кастомных наборов защищено: встроенные сеты удалить через бота нельзя
- Рекомендуется отдельный Twitter-аккаунт (не личный) для работы системы

### 8.3 Потенциальные риски

| Риск | Описание | Митигация |
|---|---|---|
| Блокировка Twitter | Автоматический сбор нарушает ToS X/Twitter | Задержки между запросами, отдельный аккаунт |
| Утечка credentials | Компрометация `.env` или сервера | Хранить `.env` вне репозитория, использовать ключи с минимальными правами |
| Ложная аналитика | LLM может ошибочно ранжировать твиты | Проверка результатов, настройка промптов |
| Расходы на API | Рост объёма → рост затрат | Мониторинг usage в Anthropic Console |

## 9. Экономика и эффект

### 9.1 Трудозатраты до/после

| Активность | До (вручную) | После (автоматически) |
|---|---|---|
| Мониторинг Twitter | 30–60 мин/день | 0 мин (автомат) |
| Чтение новостей (a16z, TC) | 20–40 мин/день | 5–10 мин (только дайджест) |
| Итого в день | **60–120 мин** | **5–15 мин** |
| Итого в месяц | **30–60 часов** | **2.5–7.5 часов** |

**Экономия: ~50–55 часов/месяц**

### 9.2 Стоимость

| Статья | Стоимость |
|---|---|
| Claude Haiku API | ~$3–6/месяц (при типичном использовании) |
| VPS (минимальный) | ~$3–5/месяц |
| **Итого** | **~$6–11/месяц** |

### 9.3 Ожидаемый эффект

- **Скорость**: время на ежедневный мониторинг сокращается в 4–10 раз
- **Качество**: LLM-фильтрация убирает «шум» — остаются только значимые материалы
- **Охват**: система мониторит источники 24/7, не пропуская публикации
- **Масштабируемость**: добавление нового источника — создать JSON-файл с аккаунтами

### 9.4 План внедрения (2–3 недели)

| Этап | Длительность | Действия |
|---|---|---|
| **1. Подготовка** | 1–2 дня | Получить API-ключи (Claude, Telegram), создать отдельный Twitter-аккаунт |
| **2. Развёртывание** | 1 день | Клонировать репозиторий, настроить `.env`, запустить бота на VPS |
| **3. Настройка** | 3–5 дней | Добавить нужные Twitter-наборы, настроить время расписания, протестировать |
| **4. Эксплуатация** | ongoing | Мониторинг работы, добавление новых источников по необходимости |
